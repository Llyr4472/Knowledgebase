Advanced File Disclosure

Not all XXE vulnerabilities may be straightforward to exploit. Some file formats may not be readable through basic XXE, while in other cases, the web application may not output any input values in some instances, so we may try to force it through errors.

Advanced Exfiltration with CDATA

In previous instances, PHP filters were used to encode PHP source files to prevent breaking the XML format. However, to extract data (including binary data) for other web applications, we can utilize CDATA tags.

To output data that does not conform to the XML format, we can wrap the content of the external file reference with a CDATA tag (e.g. <![CDATA[ FILE_CONTENT ]]>). This way, the XML parser considers this part raw data, which may contain any type of data, including special characters.

The Challenge: Internal vs External Entities

One approach might be to define a begin internal entity with <![CDATA[, an end internal entity with ]]>, and place the external entity file in between.

If we reference the &joined; entity, it should theoretically contain our escaped data. However, this will not work, since XML prevents joining internal and external entities.

The Solution: XML Parameter Entities

To bypass this limitation, we can utilize XML Parameter Entities. These start with a % character and can only be used within the DTD. If referenced from an external source (e.g., our own server), all of them are considered external and can be joined.

Note

In some modern web servers, we may not be able to read some files (like index.php), as the web server would be preventing a DOS attack caused by file/entity self-reference (i.e., XML entity reference loop).

This trick is handy when basic XXE methods do not work or when dealing with different web development frameworks.

Error Based XXE

In situations where the web application does not write any output, we cannot control XML input entities to write content. If the application displays runtime errors (e.g., PHP errors) and lacks proper exception handling, we can use these errors to read the output of the XXE exploit.

Testing for Errors

First, send malformed XML data to see if the application displays errors. This can be done by deleting closing tags, changing tags so they don’t close (e.g. <roo> instead of <root>), or referencing a non-existing entity.

Result: HTTP POST request to /error/submitDetails.php with XML data causing a ‘nonExistingEntity’ error in PHP response.

If the error reveals the web server directory, it confirms we can exploit this flaw to exfiltrate content.

Reading Source Code

This method can be used to read source code by pointing the DTD to the target file (e.g., file:///var/www/html/submitDetails.php). However, it is less reliable than previous methods due to potential length limitations and special characters breaking the output.
